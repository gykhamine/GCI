COMPRENDRE LES CAPSULES APPLICATIVES

Le guide d'architecture pour les créateurs de solutions souveraines

Qu'est-ce qu'une Capsule ?

La métaphore de la console de jeu

Pour comprendre une capsule sans jargon technique, imaginez une console de jeux de salon rétro. La console possède ses composants internes, sa carte mère et ses boutons : c'est Gykhamine OS (le noyau immuable). Seule, la console démarre mais n'affiche rien. Pour qu'elle fonctionne, vous devez y insérer une cartouche de jeu.

Une Capsule Applicative est exactement cette cartouche de jeu. C'est un simple dossier nommé Gykhamine/ que l'on glisse à la racine de la clé USB. Il contient vos scripts d'automatisation et votre application Django complète. Vous changez le dossier, vous changez instantanément l'intégralité du métier de l'ordinateur.

Anatomie d'une Capsule

Le système Gykhamine OS est programmé pour chercher un dossier spécifique au boot. Pour être valide, votre dossier capsule doit obligatoirement respecter l'arborescence et le nommage suivants :

📁 Racine_de_la_clé_USB (Partition EFI / Gykhamine Live)
└── 📁 Gykhamine/ (Le répertoire racine de votre Capsule)
    ├── 📁 1/ (Scripts d'initialisation système — Exécutés en mode administrateur SUDO)
    │   └── 📄 gy.py (Exemple : configuration de périphériques locaux)
    ├── 📁 2/ (Scripts d'initialisation utilisateur — Exécutés SANS privilège SUDO)
    │   └── 📄 gy.py (Exemple : variables d'environnement locales)
    └── 📁 my_project/ (L'arborescence de votre code métier de production)
        ├── 📄 manage.py (Le point d'entrée de votre application web Django)
        ├── 📁 my_project/ (Configuration globale du projet, settings.py, urls.py)
        └── 📁 apps_metiers/ (Vos modules Django : POS, stocks, factures, élèves...)

Le Cycle d'Exécution (Comment ça tourne ?)

Voici le cheminement logique précis effectué par l'infrastructure immuable lors du démarrage de la machine cible :

I

Amorçage immuable (Kernel & Dracut)

L'ordinateur boot sur les fichiers du répertoire live/. Le système charge un environnement minimaliste et sécurisé en lecture seule (SquashFS), garantissant l'absence totale de virus ou d'altération du système.

II

Détection et Routage Bas Niveau (Le dossier 1)

Le système monte la partition EFI et détecte le dossier Gykhamine/. Il bascule immédiatement dans le répertoire 1/ et exécute le script 1/gy.py avec les droits d'administration (Root via sudo). C'est le moment idéal pour charger des modules de noyau spécifiques, configurer le réseau local ou monter des disques durs persistants.

III

Initialisation de l'Espace Utilisateur (Le dossier 2)

le système entre dans le répertoire 2/. Il y lance le script 2/gy.pysous l'identité de l'utilisateur standard (sans privilèges). Ce routine prépare l'environnement applicatif local, vérifient l'intégrité des bases de données locales (comme SQLite) et préparent le terrain pour le service web.

IV

Instanciation et Lancement (my_project)

Enfin, le script d'autostart natif 1/gy.py cible le dossier my_project/. Il lance le serveur web (Django), configure l'écoute sur l'adresse locale et ouvre instantanément l'interface applicative en plein écran pour les opérateurs. La plateforme locale est opérationnelle et résiliente.

🛡️ Rappel d'Architecture : Puisque Gykhamine OS est immuable, si un utilisateur fait une mauvaise manipulation ou éteint brutalement l'ordinateur, aucun fichier système n'est corrompu. Au prochain redémarrage, le cycle reprend à zéro à partir de l'image ISO gravée et de votre capsule saine.
