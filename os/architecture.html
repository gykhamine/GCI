📘 Anatomie Réelle d'une Capsule

Une capsule Gykhamine n'est pas un simple dossier applicatif. C'est une unité structurée qui encapsule à la fois la logique métier, l'initialisation système, les dépenda[...]

L'organisation /Gykhamine/ n'est donc pas décorative : elle représente une frontière fonctionnelle stricte entre le système immuable et la couche dynamique métier.

Dans ce modèle, chaque capsule est découpée en sous-niveaux hiérarchiques, chacun ayant un rôle précis dans le cycle de vie du système.

1. Structure racine et logique de séparation

La capsule commence toujours dans le répertoire :

/Gykhamine/

Ce répertoire agit comme point d'entrée logique. Il est détecté automatiquement au démarrage du système. Sa présence suffit à activer le mode capsule, ce qui signifie que [...]

Le dossier /Gykhamine/ est ensuite divisé en modules numérotés :

/1/ → couche système (privilégiée)
/2/ → couche utilisateur (non privilégiée)
/my_project/ → couche métier (Django)

Cette séparation est essentielle car elle introduit une hiérarchie de privilèges claire :

/1 contrôle le système
/2 contrôle l'utilisateur
/my_project contrôle la logique métier

2. Couche système (/1) : le noyau opérationnel de la capsule

Le dossier /1 représente le cœur technique de la capsule. Il contient le script principal gy.py, qui agit comme orchestrateur du système.

Ce script n'est pas une simple automatisation. Il est responsable de la configuration complète de l'environnement machine :

initialisation des bases de données (PostgreSQL)
configuration du cache (Redis)
mise en place du réseau
gestion du firewall
déploiement de NGINX
configuration des services système
gestion du stockage persistant
supervision de l'état système

Le rôle de cette couche est de transformer un système Linux standard en plateforme applicative spécialisée.

On peut la considérer comme une "usine de préparation d'environnement".

Elle garantit que, quel que soit l'état initial de la machine, l'environnement final sera toujours conforme aux besoins de la capsule.

3. Couche utilisateur (/2) : environnement non privilégié

Le dossier /2 contient également un gy.py, mais avec une différence fondamentale : il n'a aucun accès root.

Cette couche est responsable de tout ce qui concerne l'utilisateur final et l'interaction locale :

configuration de l'interface utilisateur
mapping clavier / manette
automatisation des périphériques
outils locaux
scripts non critiques
comportements applicatifs locaux

Cette séparation est essentielle pour la sécurité du système. Elle empêche toute action utilisateur de compromettre l'infrastructure.

On peut dire que /2 représente "l'expérience utilisateur contrôlée".

4. Couche métier (/my_project) : cœur applicatif Django

Cette partie contient le projet Django principal. C'est ici que réside la logique métier réelle de la capsule.

On y trouve :

manage.py
configuration Django (settings.py, urls.py, etc.)
applications métiers

Les applications métiers sont organisées en modules indépendants :

POS (point de vente)
stocks
facturation
gestion élèves
monitoring

Cette architecture permet une séparation stricte des domaines fonctionnels. Chaque module peut évoluer indépendamment, tout en restant intégré dans la capsule globale.

5. Principe de cohérence interne

Une capsule Gykhamine doit respecter une règle fondamentale : elle doit être auto-suffisante.

Cela signifie qu'elle doit contenir tout ce qui est nécessaire à son exécution :

services système
dépendances logicielles
logique métier
règles de déploiement
scripts d'initialisation

Une capsule ne dépend pas d'un environnement externe instable. Elle construit son propre environnement sur une base minimale fournie par le système immuable.

6. Principe de portabilité contrôlée

Même si la capsule est autonome, elle reste portable.

Cela signifie qu'un simple déplacement du dossier /Gykhamine/ permet de reproduire exactement le même système sur une autre machine.

Cependant, cette portabilité n'est pas anarchique : elle est contrôlée par les scripts d'initialisation (gy.py), qui reconstruisent l'environnement de manière déterminist[...]

7. Vision architecturale globale

L'anatomie d'une capsule n'est pas seulement une structure de fichiers. C'est un modèle de pensée :

le système est stable et immuable
la capsule est dynamique et remplaçable
chaque couche a un rôle strictement défini
les privilèges sont séparés par conception
l'environnement est reconstruit automatiquement

Cette architecture permet de transformer une machine classique en infrastructure spécialisée capable de changer totalement de fonction sans réinstallation.

📘 Cycle d'Exécution Réel des Capsules Gykhamine

Le cycle d'exécution d'une capsule Gykhamine décrit la manière dont une machine passe d'un simple état système (Linux immuable) à un état fonctionnel complet dirigé pa[...]

Dans Gykhamine, le démarrage ne consiste pas à "lancer des logiciels", mais à construire un environnement cohérent autour d'une capsule détectée.

Ce cycle est découpé en quatre grandes étapes logiques.

1. Étape I — Amorçage du système immuable

Le système démarre depuis un environnement minimal situé dans :

/run/initramfs/live

À ce stade, la machine est dans un état neutre. Aucun service métier n'est encore actif. Le système fonctionne sur une base en lecture seule (souvent via SquashFS ou équivale[...]

Cette immuabilité joue un rôle critique :

elle empêche la corruption du système
elle garantit la reproductibilité du démarrage
elle assure une base stable indépendante des erreurs humaines
elle limite les effets des logiciels malveillants persistants

Dans cette phase, le système est comparable à une "plateforme vide sécurisée", prête à recevoir une définition fonctionnelle.

Aucune logique métier n'est encore active. Seuls les services essentiels du noyau et de l'environnement de base sont chargés.

2. Étape II — Détection et activation de la capsule

Une fois le système stabilisé, un processus de détection est lancé pour identifier la présence de la structure :

/Gykhamine/

Ce dossier est le déclencheur fondamental du mode capsule.

Dès sa détection, le système considère que la machine ne fonctionne plus en mode générique, mais en mode spécialisé.

Le système entre alors dans :

/Gykhamine/1/

et exécute automatiquement :

gy.py

avec privilèges administrateur.

Ce moment est critique : il marque le passage entre système passif et infrastructure active.

Le script /1/gy.py devient alors l'orchestrateur du système. Il ne se contente pas de lancer des services : il reconstruit l'environnement complet.

3. Étape III — Initialisation de la couche utilisateur

Une fois l'infrastructure système configurée, le cycle passe à une deuxième phase :

/Gykhamine/2/

Ici est exécuté :

gy.py (sans privilèges root)

Cette étape est volontairement séparée de la couche système pour garantir une isolation stricte.
